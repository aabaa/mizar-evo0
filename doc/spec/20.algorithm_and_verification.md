Mizar Evolution integrates **executable algorithms** with the formal proof system. An `algorithm` block defines a computation that can be:

- **Run** by the Mizar Virtual Machine (MVM) during verification (`by computation`).
- **Reasoned about** using Hoare-logic contracts (`requires`, `ensures`, `invariant`).
- **Promoted** to a mathematical functor (`func`) when total correctness is established.
- **Extracted** to an executable host-language program.

The design follows the principle of **partial correctness by default**: an algorithm is permitted to diverge unless an explicit termination argument is given. This avoids demanding termination proofs for algorithms that are only ever used on concrete inputs or in bounded contexts.

## 20.1 Algorithm Syntax

### 20.1.1 Algorithm Definition Block

An algorithm is declared inside a `definition` block using the `algorithm` keyword. The surrounding `let` declarations bind the type context, and the algorithm's parameter list selects which of those bindings become formal parameters.

```ebnf
algorithm_def ::= "algorithm" identifier "(" [ identifier_list ] ")" "->" type_expression
                  { algorithm_contract }
                  algorithm_body ";" ;

identifier_list  ::= identifier { "," identifier } ;

algorithm_contract ::= "requires" formula
                     | "ensures" formula ;

algorithm_body   ::= "do" algo_statement_list "end" ;
```

```mizar
definition
  let n be Nat;

  algorithm Factorial(n) -> Nat
    requires n >= 0
    ensures result >= 1
  do
    decreasing n;
    if n = 0 do return 1; end;
    return n * Factorial(n - 1);
  end;
end;
```

An algorithm defined at module top-level (outside any `definition` block) must carry its own parameter declarations with types:

```ebnf
standalone_algorithm_def ::= "algorithm" identifier
                             "(" [ typed_param_list ] ")" "->" type_expression
                             { algorithm_contract }
                             algorithm_body ";" ;

typed_param_list ::= typed_param { "," typed_param } ;
typed_param      ::= identifier ":" type_expression ;
```

### 20.1.2 Parameters and Return Type

Parameters are positional and immutable within the algorithm body (they behave as `const`). To use a mutable version of a parameter, create a `var` copy:

```mizar
algorithm CopyAndModify(x) -> Nat
do
  var y := x;
  y := y + 1;
  return y;
end;
```

The return type specifies the type of the value returned by every `return` statement. The special identifier `result` in `ensures` clauses refers to the return value:

```mizar
algorithm Abs(n) -> Nat
  ensures result >= 0
do
  if n >= 0 do return n; end;
  return -n;
end;
```

### 20.1.3 Variable Declarations (`var`, `const`)

Inside an algorithm body, variables are introduced with `var` (mutable) or `const` (immutable):

```ebnf
var_decl   ::= "var" identifier [ ":" type_expression ] ":=" expr ";" ;
const_decl ::= "const" identifier [ ":" type_expression ] "=" expr ";" ;
```

| Keyword | Mutability | Re-assignable | Use case |
|---|---|---|---|
| `const` | Immutable | No | Intermediate results, captured values, `the Element of S` picks |
| `var` | Mutable | Yes (`:=`) | Loop counters, accumulators, state |

```mizar
algorithm Sum(n) -> Nat
do
  var acc := 0;
  for i = 1 to n do
    acc := acc + i;
  end;
  return acc;
end;
```

If the type annotation is omitted, the type is inferred from the initializer expression (see [Chapter 8, Type Inference](./08.type_inference.md)).

### 20.1.4 Assignment

Mutable variables are updated with `:=`:

```ebnf
assignment ::= lvalue ":=" expr ";" ;
lvalue     ::= identifier
             | lvalue "." identifier
             | lvalue "[" expr "]" ;
```

```mizar
var counter := 0;
counter := counter + 1;           :: scalar update
state.value := state.value + 1;   :: field update
arr[i] := arr[i] * 2;             :: indexed update (requires arr to be a mutable array)
```

Only `var`-declared identifiers and their sub-fields/indices are valid lvalues. Assigning to a `const` identifier or a `let`-bound variable is a **compile-time error**.

## 20.2 Control Flow

### 20.2.1 Conditional (`if`)

```ebnf
if_stmt ::= "if" formula "do"
              algo_statement_list
            [ "else" ( if_stmt | "do" algo_statement_list "end" ) ]
            "end" ";" ;
```

```mizar
if n = 0 do
  return 1;
end;

if n mod 2 = 0 do
  return n / 2;
else do
  return 3 * n + 1;
end;

:: if-else-if chain
if n < 0 do
  return -1;
else if n = 0 do
  return 0;
else do
  return 1;
end;
```

The condition is a formula (predicate). Both branches must return the same type when `return` is used, or both must fall through to subsequent statements.

### 20.2.2 Bounded Loop (`while`)

```ebnf
while_stmt ::= "while" formula "do"
                 { loop_annotation ";" }
                 algo_statement_list
               "end" ";" ;

loop_annotation ::= "invariant" formula
                  | "decreasing" term_list [ "wrt" identifier ] ;
```

Loop annotations (`invariant`, `decreasing`) appear at the **top** of the `while` body, before any other statements. They are optional unless the verifier requires them for a specific proof goal.

```mizar
var i := 1;
var sum := 0;
while i <= n do
  invariant sum = i * (i - 1) / 2 & i >= 1;
  decreasing n - i + 1;
  sum := sum + i;
  i := i + 1;
end;
```

### 20.2.3 Range Loop (`for ... to`)

```ebnf
for_range_stmt ::= "for" identifier "=" expr ( "to" | "downto" ) expr
                   [ "step" expr ]
                   "do" algo_statement_list "end" ";" ;
```

The loop variable `identifier` is immutable within the loop body. Its scope is limited to the loop body. The bounds and step are evaluated once before the loop begins.

- `for i = 1 to n` — iterates i = 1, 2, ..., n (inclusive, ascending).
- `for i = n downto 1` — iterates i = n, n−1, ..., 1 (inclusive, descending).
- `step s` — increment/decrement by s each iteration (s must be positive).

```mizar
:: Sum of first n natural numbers
var sum := 0;
for i = 1 to n do
  sum := sum + i;
end;

:: Countdown
for i = 10 downto 1 do
  process(i);
end;
```

Termination of `for ... to` loops is **automatically guaranteed** by the verifier — the loop count is determined at entry, so no `decreasing` annotation is needed.

### 20.2.4 Collection Loop (`for ... in`)

```ebnf
for_collection_stmt ::= "for" identifier "in" expr
                        "do" algo_statement_list "end" ";" ;
```

Iterates over all elements of a **finite** collection. The verifier requires the collection to be a `finite set` (or a type registering `finite`); this is a proof obligation on the iteration site.

```mizar
:: Compute product of all elements of a finite set of Nats
var product := 1;
for x in S do
  product := product * x;
end;
```

The iteration order is implementation-defined; the verifier treats the loop semantically as a fold over the set. Do not rely on any particular order.

### 20.2.5 Pattern Match (`match`)

```ebnf
match_stmt ::= "match" expr "with"
               match_case+
               ( "otherwise" "do" algo_statement_list "end" ";"
               | exhaustiveness_proof )
               "end" ";" ;

match_case          ::= "case" formula "do" algo_statement_list "end" ";" ;
exhaustiveness_proof ::= "exhaustive" [ "by" reference_list ] ";" ;
```

The cases are tested in order; the first case whose formula holds is executed, and the rest are skipped.

**Exhaustiveness is mandatory.** A `match` without `otherwise` must end with `exhaustive`, supplying a proof that the disjunction of all case predicates covers every possible value of the matched expression. Omitting both `otherwise` and `exhaustive` is a **compile-time error**. This rule prevents silent fall-through and keeps exhaustiveness checking decidable.

```mizar
:: With otherwise — handles any remaining case unconditionally
match n with
  case n > 0 do return  1; end;
  case n < 0 do return -1; end;
  otherwise do return 0; end;
end;

:: With exhaustive proof — no otherwise needed when the verifier
:: can confirm the cases cover all Integer values
match n with
  case n > 0 do return  1; end;
  case n < 0 do return -1; end;
  case n = 0 do return  0; end;
  exhaustive by integer_trichotomy;
end;

:: Type-based dispatch (shorthand for formula x is T)
match x with
  case x is NatNumber do
    return handle_nat(x);
  end;
  case x is NegInteger do
    return handle_neg(x);
  end;
  otherwise do
    return handle_other(x);
  end;
end;
```

> [!NOTE]
> `case x is T` is syntactic sugar for `case x : T holds` (a type assertion formula). Cases may overlap — the **first** matching case wins. The verifier does not require cases to be disjoint, but exhaustiveness (whether via `otherwise` or `exhaustive`) is always required.

### 20.2.6 Jump Statements (`return`, `break`, `continue`)

```ebnf
return_stmt   ::= "return" [ expr ] ";" ;
break_stmt    ::= "break" ";" ;
continue_stmt ::= "continue" ";" ;
```

| Statement | Effect | Verification obligation |
|---|---|---|
| `return expr` | Exit algorithm with value `expr` | `expr` must satisfy the `ensures` postcondition |
| `return` | Exit a void algorithm (no return type) | All `ensures` conditions must hold |
| `break` | Exit the immediately enclosing loop | Loop `invariant` must hold at break point |
| `continue` | Skip to the next loop iteration | Loop `invariant` must hold at continue point |

```mizar
algorithm FirstEven(S : finite set of Integer) -> Integer
  ensures result = -1 or result mod 2 = 0
do
  for x in S do
    if x mod 2 = 0 do
      return x;   :: exits algorithm; ensures checked here
    end;
  end;
  return -1;      :: sentinel: no even element found
end;
```

## 20.3 Non-determinism (Pick)

A **Pick** operation selects an arbitrary element from a non-empty set. It is expressed using Mizar's existing `the` construct:

```ebnf
pick_expr ::= "the" type_expression "of" expr ;
            | "the" type_expression ;     (* arbitrary element of the type *)
```

```mizar
const x = the Element of S;     :: pick any element of S
const p = the prime Number;      :: pick any prime (type-level pick)
```

**Semantics:**

- The MVM selects one element of `S` non-deterministically at runtime. The choice may differ across executions.
- The verifier treats `the Element of S` as *some* element satisfying `it in S` — existential, not specific.
- If `S` is empty, execution raises a **runtime verification failure** (see §20.8.3).

**Use in proofs:** When `by computation` is used to verify a goal, Pick operations are resolved by the MVM choosing a concrete witness. The verification succeeds if *some* choice satisfies the goal. If the goal must hold for *every* choice, `by computation` alone is insufficient — a classical proof is required.

```mizar
:: Non-deterministic search for an even element
algorithm FindEven(S) -> Integer
do
  var current := S;
  while current <> {} do
    invariant current c= S;
    decreasing card current;
    const x = the Element of current;
    if x mod 2 = 0 do
      return x;
    end;
    current := current \ {x};
  end;
  return -1;
end;
```

> [!NOTE]
> Pick on an infinite set is permitted in algorithm definitions. However, the MVM cannot execute Pick on infinite sets — doing so raises a runtime verification failure (§20.8.3). Such algorithms may still be used in proofs where only their logical specification (contracts) matters.

## 20.4 Contracts (`requires`, `ensures`, `assert`)

Mizar Evolution adopts **Hoare-logic** style contracts embedded directly in algorithm definitions. Contracts serve as the interface between the algorithm's operational behaviour and the surrounding formal proof system.

### 20.4.1 Preconditions and Postconditions

```ebnf
algorithm_contract ::= "requires" formula    (* precondition *)
                     | "ensures"  formula ;  (* postcondition *)
```

- `requires P` — states that `P` must hold when the algorithm is called. The verifier checks `P` at every call site. The algorithm body may assume `P`.
- `ensures Q` — states that `Q` holds upon normal termination. `result` in `Q` refers to the return value.

Multiple `requires` and `ensures` clauses may appear; they are treated as conjunctions:

```mizar
algorithm Divide(a, b) -> Rational
  requires b <> 0
  ensures result * b = a
do
  return a / b;
end;
```

Contracts are **proof obligations**:
- At each call site `Divide(x, y)`, the verifier demands a proof that `y <> 0`.
- Within the algorithm body, after `return v`, the verifier demands a proof that `v * b = a`.

If the proof cannot be discharged automatically, an explicit proof block may follow the contract:

```mizar
algorithm Sqrt(x) -> Real
  requires x >= 0
  proof
    :: justify the precondition discharge at call sites
    thus x >= 0;
  end
  ensures result >= 0 & result * result = x
do
  ...
end;
```

### 20.4.2 Inline Assertions

```ebnf
assert_stmt ::= "assert" formula [ "by" reference_list ] ";" ;
```

`assert P` inserts a proof checkpoint: the verifier must establish `P` at that program point, using the current logical context (all `requires`, `invariant`, and prior `assert` conditions in scope, plus any `by`-cited lemmas).

```mizar
algorithm BinarySearch(arr, lo, hi, target) -> Integer
do
  invariant lo <= hi + 1;
  ...
  const mid := (lo + hi) / 2;
  assert lo <= mid & mid <= hi by arithmetic;
  ...
end;
```

If `assert` fails to verify, the verifier reports a **contract violation** error and halts verification at that point.

### 20.4.3 Verification Obligations

The table below summarises all obligations generated by algorithm verification:

| Construct | Obligation |
|---|---|
| Call site of `f(args)` | `requires` preconditions of `f`, instantiated with `args` |
| `return expr` | `ensures` postconditions, with `result` bound to `expr` |
| `assert P` | `P` holds at that program point |
| `break` inside `while` | loop `invariant` holds at break |
| `continue` inside `while` | loop `invariant` holds at continue |
| `for x in S` | `S` is `finite` |
| `the Element of S` | `S <> {}` (non-emptiness) |
| `decreasing e` | `e` decreases strictly on each iteration / recursive call |
| `totality` | all `decreasing` measures are valid well-founded, algorithm returns on all inputs |

## 20.5 Loop Verification

### 20.5.1 Loop Invariants (`invariant`)

A loop invariant is a formula that must hold:
1. **Before** the first iteration (establishment).
2. **After** every completed iteration (maintenance).
3. **On exit** (whether by condition becoming false, `break`, or `continue`).

```mizar
var i := 0;
var sum := 0;
while i < n do
  invariant 0 <= i & i <= n & sum = i * (i + 1) / 2;
  decreasing n - i;
  i := i + 1;
  sum := sum + i;
end;
:: After loop: i = n & sum = n * (n + 1) / 2
```

Multiple `invariant` clauses are allowed and are interpreted as a conjunction:

```mizar
while cond do
  invariant P;
  invariant Q;
  ...
end;
```

**Establishment** is checked at the point where the loop is first entered. **Maintenance** is checked by the verifier assuming the invariant holds at the loop head and verifying it holds at the end of the body.

### 20.5.2 Termination Measure (`decreasing`)

The `decreasing` annotation declares a **termination measure**: an expression that decreases on every loop iteration (or recursive call). The verifier generates an obligation that the measure decreases and that it is bounded below.

```ebnf
termination_decl ::= "decreasing" term_list [ "wrt" identifier ] ;
term_list        ::= term_expression { "," term_expression } ;
```

| Form | Ordering used | Example |
|---|---|---|
| `decreasing e` | `<` on `Nat` (standard well-order) | `decreasing n - i` |
| `decreasing e, f` | Lexicographic `<` on pairs | `decreasing m, n` |
| `decreasing e wrt R` | Well-founded relation `R` | `decreasing x wrt SubsetOrder` |

For `decreasing e` with a `Nat`-valued expression, the standard ordering `<` on natural numbers is used. For tuple form, lexicographic order is used; the tuple decreases if the leftmost differing component decreases (later components need not decrease if an earlier one does).

The general form `decreasing e wrt R` requires the verifier to confirm that:
1. `R` is a well-founded relation (registered via a cluster or declared theorem).
2. On each iteration, `(e_new, e_old) in R`.

## 20.6 Termination Proofs

### 20.6.1 Declaring a Termination Measure

For a recursive algorithm, `decreasing` appears at the **top** of the algorithm body (before any other statements):

```mizar
definition
  let m, n be Nat;

  algorithm Ackermann(m, n) -> Nat
  do
    decreasing m, n;    :: lexicographic decrease on (m, n)

    if m = 0 do return n + 1; end;
    if n = 0 do return Ackermann(m - 1, 1); end;
    return Ackermann(m - 1, Ackermann(m, n - 1));
  end;
end;
```

The verifier checks that on each recursive call, the tuple `(m, n)` decreases in lexicographic order with respect to `<`:

- `Ackermann(m - 1, 1)` — first component `m - 1 < m`. ✓
- `Ackermann(m, n - 1)` — first components equal, second `n - 1 < n`. ✓
- `Ackermann(m - 1, Ackermann(m, n - 1))` — first component `m - 1 < m`. ✓

### 20.6.2 Well-Founded Relations

Custom well-founded relations can be used when the standard lexicographic order is insufficient:

```mizar
definition
  let T be set;
  let R be Relation of T, T;

  :: Declare R as well-founded
  :: (requires cluster or theorem registration)
  algorithm DescendR(x) -> T
  do
    decreasing x wrt R;
    ...
  end;
end;
```

The verifier looks up whether `R` is registered as well-founded (via a `cluster ... for Relation` registration or a preceding theorem `R is well_founded`). If not registered, a **missing well-foundedness** error is reported.

### 20.6.3 Totality and Promotion to `func`

An algorithm with a verified termination measure can be **promoted** to a mathematical functor using the `totality` correctness condition. This makes the algorithm usable everywhere a `func` is expected — not just in `by computation` proofs, but in arbitrary formal reasoning.

```ebnf
totality_condition ::= "totality" [ proof_block ] ;
```

```mizar
definition
  let n be Nat;

  algorithm Factorial(n) -> Nat
  do
    decreasing n;
    if n = 0 do return 1; end;
    return n * Factorial(n - 1);
  end;
  totality;
  :: Verification obligation: the algorithm terminates for every Nat n,
  :: i.e., the decreasing measure n is a valid well-founded argument
  :: and every recursive call satisfies the requires preconditions.
end;
```

Once `totality` is declared and verified, `Factorial` may be used as a functor:

```mizar
theorem : Factorial(0) = 1;
theorem : Factorial(n + 1) = (n + 1) * Factorial(n);
```

> [!NOTE]
> Without `totality`, the algorithm exists only as a computable procedure. It can be called with `by computation` for concrete values, but it cannot be universally quantified over or used as a first-class functor in proofs.

## 20.7 Recursive Definitions

### 20.7.1 Direct Recursion

An algorithm may call itself recursively. The verifier accepts recursive calls without additional annotation, but requires a `decreasing` measure to prove termination (§20.6). Without `decreasing`, the algorithm is treated as partially defined — termination is not guaranteed.

```mizar
algorithm Fibonacci(n) -> Nat
do
  decreasing n;
  if n = 0 do return 0; end;
  if n = 1 do return 1; end;
  return Fibonacci(n - 1) + Fibonacci(n - 2);
end;
```

Recursive calls must satisfy the `requires` preconditions of the algorithm at each call site. The verifier generates proof obligations for each recursive call's preconditions.

### 20.7.2 Mutual Recursion

Two or more algorithms may call each other mutually. They must be declared within the same `definition` block (or within a `mutual` grouping):

```ebnf
mutual_block ::= "mutual" "do"
                   { algorithm_def }
                 "end" ";" ;
```

```mizar
definition
  let n be Nat;

  mutual do
    algorithm IsEven(n) -> bool
    do
      decreasing n;
      if n = 0 do return true; end;
      return IsOdd(n - 1);
    end;

    algorithm IsOdd(n) -> bool
    do
      decreasing n;
      if n = 0 do return false; end;
      return IsEven(n - 1);
    end;
  end;
end;
```

For mutual recursion, the `decreasing` measures must collectively establish termination — typically by a shared measure that decreases on every call in any cycle. The verifier checks that no infinite call cycle can occur.

## 20.8 Execution Model and Computability

### 20.8.1 The Mizar Virtual Machine (MVM)

The **Mizar Virtual Machine (MVM)** is the runtime component that evaluates algorithm expressions during verification. The MVM is invoked when:

- A `by computation` proof step is used (§20.8.2).
- An algorithm is called with fully concrete (ground) arguments in a verification context.

The MVM operates on a **call-by-value** model: arguments are fully evaluated before a call is made. Side effects are confined to local mutable state (`var` variables); there is no global mutable state.

### 20.8.2 `by computation` Proof Step

The `by computation` proof step instructs the MVM to verify the current goal by evaluation:

```ebnf
computation_proof ::= "by" "computation" [ "[" computation_option { "," computation_option } "]" ] ;
computation_option ::= "steps" "=" nat_literal
                     | "timeout" "=" nat_literal ;
```

```mizar
theorem Fact5 : Factorial(5) = 120
proof
  thus thesis by computation;
end;

:: Short form
theorem : Factorial(5) = 120 by computation;
```

**What `by computation` can prove:**

| Goal form | MVM behaviour |
|---|---|
| `f(v₁, ..., vₙ) = c` (ground equality) | Evaluate `f(v₁, ..., vₙ)`, check result equals `c` |
| `P(v₁, ..., vₙ)` (ground predicate) | Evaluate `P`, check it holds |
| `ex x st P(x)` (existential) | Run search; if a witness `w` is found, **verify** `P(w)` holds before accepting |
| Non-ground formulas | **Not supported** — `by computation` requires ground terms |

For non-ground goals, use classical proofs. `by computation` is strictly a tool for **concrete evaluation**.

**Witness verification.** For existential goals, finding a witness is not sufficient — the MVM must also formally verify that the candidate witness satisfies the goal predicate. This two-step process (search + verify) ensures soundness regardless of the search strategy:

```
by computation for  ex x st P(x) :
  1. MVM runs and produces candidate w.
  2. MVM evaluates P(w) and confirms it holds.
  3. Only if step 2 succeeds does the proof step succeed.
```

If step 2 fails (the witness does not actually satisfy `P`), the proof step fails with a **witness verification failure** error, not a silent success. This guarantees that `by computation` never produces an unsound proof even if the search heuristic returns an incorrect candidate.

### 20.8.3 Computability and Runtime Failures

Algorithm definitions are permitted to call non-constructive operations (non-computable functions, Pick on infinite sets, etc.). Such definitions are mathematically valid but the MVM cannot execute them.

When the MVM encounters an unexecutable operation, it raises a **runtime verification failure**:

| Operation | Runtime failure condition |
|---|---|
| `the Element of S` | `S` is infinite or its enumeration is unavailable |
| Call to `func f(x)` without a `algorithm` implementation | `f` has no computable body |
| Division by zero (`a / 0`) | Zero denominator |
| Out-of-bounds array access | Index out of range |

A runtime verification failure causes the `by computation` proof step to **fail** — but it does not invalidate the algorithm's definition. The definition remains mathematically meaningful; only its executability via MVM is limited.

> [!NOTE]
> Runtime failures are **not** compile-time errors. An algorithm may legally reference non-computable functions for mathematical generality; the restriction only applies when the MVM is asked to execute it.

### 20.8.4 Gas Limit

To prevent non-terminating algorithms from hanging the verifier, the MVM enforces a **gas limit**: a maximum number of execution steps per `by computation` invocation. When the limit is exceeded, verification fails with a **timeout** error.

The default gas limit is implementation-defined. It can be overridden locally:

```mizar
theorem : HardComputation(n) = expected_value
proof
  thus thesis by computation[steps = 1000000];
end;
```

Setting `steps = 0` disables the gas limit (use with caution). The `timeout` option sets a wall-clock timeout in milliseconds as an alternative bound.

## 20.9 Code Extraction

Mizar Evolution can extract algorithms to executable programs in a supported host language. Extraction produces a **verified** program: the extracted code preserves the semantics of the Mizar algorithm, so any property proved in Mizar holds for the extracted code by construction.

**Target languages** (in order of planned availability):

| Language | Primary use cases |
|---|---|
| Python | Rapid prototyping, cryptographic protocol scripts, quantum circuit simulation (Qiskit, Cirq) |
| C | High-performance cryptographic implementations, embedded systems, FFI entry points |
| Rust | Memory-safe systems code, production cryptography (RustCrypto), WebAssembly targets |

The intended workflow is:

```
Mizar formal verification
       ↓
algorithm + totality proof
       ↓  @[extract]
Python (prototype & test)
       ↓  @[extract(lang = "c")]
C / Rust (production deployment)
```

Typical application domains include **cryptographic protocol verification** (prove security properties in Mizar, extract to C/Rust for deployment) and **quantum algorithm verification** (prove correctness classically in Mizar, extract circuit description to Python/Qiskit for execution on real hardware).

### 20.9.1 Extraction Prerequisites

An algorithm is **extractable** only when all of the following hold:

1. **Total**: The algorithm has a verified `totality` condition (§20.6.3), guaranteeing termination.
2. **Constructive**: All called algorithms are themselves extractable, or have explicit host-language implementations supplied via `@[extract_as]` (see §20.9.2).
3. **Deterministic**: The algorithm contains no unresolved Pick operations (or all Pick operations are on finite sets with a deterministic enumeration strategy; see §20.9.3).
4. **No MVM-only operations**: The algorithm does not rely on features that have no host-language equivalent (e.g., Pick on infinite sets).

### 20.9.2 The `@[extract]` Annotation

Mark an algorithm for extraction with `@[extract]`. The optional `lang` parameter selects the target language; the default is `"python"`.

```ebnf
extract_attr ::= "@[extract" [ "(" extract_option { "," extract_option } ")" ] "]" ;
extract_option ::= "lang" "=" string_literal
                 | "module" "=" string_literal ;
```

```mizar
@[extract]
algorithm Fibonacci(n : Nat) -> Nat
do
  decreasing n;
  if n = 0 do return 0; end;
  if n = 1 do return 1; end;
  return Fibonacci(n - 1) + Fibonacci(n - 2);
end;
totality;
```

The extractor generates a corresponding function. For Python (default):

```python
# Auto-generated from Mizar algorithm Fibonacci
def fibonacci(n: int) -> int:
    if n == 0:
        return 0
    if n == 1:
        return 1
    return fibonacci(n - 1) + fibonacci(n - 2)
```

For C (`@[extract(lang = "c")]`):

```c
/* Auto-generated from Mizar algorithm Fibonacci */
uint64_t fibonacci(uint64_t n) {
    if (n == 0) return 0;
    if (n == 1) return 1;
    return fibonacci(n - 1) + fibonacci(n - 2);
}
```

**Numeric type mapping.** Mizar's unbounded `Nat` and `Integer` are mapped to the smallest host-language type that fits the statically known range (derived from `requires` preconditions). When no bound is provable, the extractor uses a bignum type (Python `int`, C `__int128` or a bignum library, Rust `num::BigUint`).

To supply a hand-written implementation for a non-extractable dependency, use `@[extract_as]`:

```mizar
@[extract_as(lang = "python", symbol = "secrets.choice")]
algorithm PickElement(S : finite set of Nat) -> Nat
  requires S <> {}
do
  return the Element of S;
end;
```

This tells the extractor to call `secrets.choice` (Python standard library) instead of generating code for that algorithm. The `lang` parameter may be repeated to supply bindings for multiple targets.

### 20.9.3 Non-determinism in Extracted Code

Pick operations on finite sets are extracted using a **deterministic strategy**, specified via `@[extract_pick]`. If no strategy is given, the extractor reports an **unresolved Pick** error rather than silently choosing one — determinism in the extracted code must be explicit.

```ebnf
extract_pick_attr ::= "@[extract_pick" "(" pick_strategy ")" "]" ;
pick_strategy     ::= string_literal ;   (* "min", "max", "first", or custom *)
```

```mizar
@[extract_pick("min")]    :: pick the minimum element
@[extract]
algorithm FindAny(S : finite set of Nat) -> Nat
  requires S <> {}
do
  return the Element of S;
end;
```

Built-in strategies:

| Strategy | Semantics |
|---|---|
| `"min"` | Minimum element under the natural ordering |
| `"max"` | Maximum element under the natural ordering |
| `"first"` | First element in the set's internal iteration order |

For custom strategies (e.g., random sampling in a cryptographic context), supply `@[extract_as]` pointing to the appropriate host-language primitive.

## 20.10 Complete Syntax (EBNF)

```ebnf
(* ── Algorithm definition ─────────────────────────────── *)
algorithm_def ::= "algorithm" identifier "(" [ identifier_list ] ")" "->" type_expression
                  { algorithm_contract }
                  algorithm_body ";" ;

standalone_algorithm_def ::= "algorithm" identifier
                             "(" [ typed_param_list ] ")" "->" type_expression
                             { algorithm_contract }
                             algorithm_body ";" ;

typed_param_list  ::= typed_param { "," typed_param } ;
typed_param       ::= identifier ":" type_expression ;

identifier_list   ::= identifier { "," identifier } ;

algorithm_contract ::= "requires" formula
                     | "ensures"  formula ;

algorithm_body    ::= "do" algo_statement_list "end" ;

mutual_block      ::= "mutual" "do" { algorithm_def } "end" ";" ;

totality_condition ::= "totality" [ proof_block ] ;

(* ── Statement list ────────────────────────────────────── *)
algo_statement_list ::= { algo_statement } ;

algo_statement ::= var_decl
                 | const_decl
                 | assignment
                 | if_stmt
                 | while_stmt
                 | for_range_stmt
                 | for_collection_stmt
                 | match_stmt
                 | return_stmt
                 | break_stmt
                 | continue_stmt
                 | assert_stmt
                 | termination_decl ";"
                 | loop_invariant ";" ;

(* ── Declarations and assignment ───────────────────────── *)
var_decl   ::= "var"   identifier [ ":" type_expression ] ":=" expr ";" ;
const_decl ::= "const" identifier [ ":" type_expression ] "="  expr ";" ;
assignment ::= lvalue ":=" expr ";" ;
lvalue     ::= identifier
             | lvalue "." identifier
             | lvalue "[" expr "]" ;

(* ── Control flow ──────────────────────────────────────── *)
if_stmt ::= "if" formula "do"
              algo_statement_list
            [ "else" ( if_stmt | "do" algo_statement_list "end" ) ]
            "end" ";" ;

while_stmt ::= "while" formula "do"
                 { loop_annotation ";" }
                 algo_statement_list
               "end" ";" ;

loop_annotation ::= loop_invariant | termination_decl ;
loop_invariant  ::= "invariant" formula ;
termination_decl ::= "decreasing" term_list [ "wrt" identifier ] ;
term_list        ::= term_expression { "," term_expression } ;

for_range_stmt ::= "for" identifier "=" expr ( "to" | "downto" ) expr
                   [ "step" expr ]
                   "do" algo_statement_list "end" ";" ;

for_collection_stmt ::= "for" identifier "in" expr
                        "do" algo_statement_list "end" ";" ;

match_stmt ::= "match" expr "with"
               match_case+
               ( "otherwise" "do" algo_statement_list "end" ";"
               | exhaustiveness_proof )
               "end" ";" ;

match_case           ::= "case" formula "do" algo_statement_list "end" ";" ;
exhaustiveness_proof ::= "exhaustive" [ "by" reference_list ] ";" ;

return_stmt   ::= "return" [ expr ] ";" ;
break_stmt    ::= "break" ";" ;
continue_stmt ::= "continue" ";" ;
assert_stmt   ::= "assert" formula [ "by" reference_list ] ";" ;

(* ── Proof step ─────────────────────────────────────────── *)
computation_proof   ::= "by" "computation"
                        [ "[" computation_option { "," computation_option } "]" ] ;
computation_option  ::= "steps"   "=" nat_literal
                      | "timeout" "=" nat_literal ;

(* ── Extraction annotations ─────────────────────────────── *)
extract_attr      ::= "@[extract" [ "(" extract_option { "," extract_option } ")" ] "]" ;
extract_option    ::= "lang"   "=" string_literal
                    | "module" "=" string_literal ;

extract_as_attr   ::= "@[extract_as" "(" extract_as_option { "," extract_as_option } ")" "]" ;
extract_as_option ::= "lang"   "=" string_literal
                    | "symbol" "=" string_literal ;

extract_pick_attr ::= "@[extract_pick" "(" string_literal ")" "]" ;

(* ── Pick expression (non-determinism) ─────────────────── *)
pick_expr ::= "the" type_expression "of" expr
            | "the" type_expression ;
```
