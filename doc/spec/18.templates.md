## 18.1 Overview

Templates are the **generics mechanism** — a facility for writing mathematical definitions and theorem schemas parameterized by types, predicates, or functions. A template captures a pattern of mathematical reasoning that is valid across a family of instances.

Templates serve two distinct roles:

1. **Parameterized Definitions**: Define attributes, modes, structures, predicates, and functors over abstract type parameters. For example, a generic `VectorSpace[K]` template can be instantiated for real numbers, complex numbers, or any field K; a `PolyRing[R]` template generates the polynomial ring over any commutative ring R.

2. **Theorem Schemas (Schemes)**: State theorems parameterized by predicates or functions — the primary mechanism for induction, recursion, and universal structural properties. These subsume the classical Mizar `scheme` construct.

**When templates are needed**

| Situation | Approach |
|---|---|
| Theorem whose type parameters are independent of any definition in the block | Ordinary universally quantified theorem — no template needed |
| Theorem co-defined with parameterized definitions (e.g., a theorem about `VectorSpace[K]` in the same block) | `theorem` item inside the template `definition` block, sharing the enclosing `let` type parameters |
| Definition (struct, mode, attr, func, pred) parameterized by a type | Template (`definition` block with `let` type parameters) |
| Theorem parameterized by a predicate or functor | Template with `let P be pred(T);` or `let F be func(T) -> S;` |

**Relationship to the `of`/`over` Syntax**

The `of`/`over` parameterization in modes (Chapter 7) and structures (Chapter 5) is **syntactic sugar** for type templates. Both single-parameter and multi-parameter forms are supported (`Subset of X` for one parameter, `Function of X, Y` for two). Both notations are accepted:

```mizar
:: Shorthand (Chapter 7 syntax)
definition
  mode Subset of X is X-subset_like set;
end;

:: Equivalent full template form
definition
  let X be type;
  mode Subset[X] is X-subset_like set;
  :: Automatic synonym: Subset of X
end;
```

The `of`/`over` shorthand is convenient for simple cases. The full `[...]` template form is equally valid and avoids syntactic ambiguity in multi-parameter cases (e.g., distinguishing type arguments from term arguments); constrained templates require the full form.

**Relationship to Classical Mizar Schemes**

Classical Mizar `scheme` declarations are unified with the general template system. A `definition` block containing `let P be pred(T);` and a `theorem` item subsumes the `scheme` mechanism using consistent syntax.

> [!NOTE]
> For hints that guide the verifier when applying template theorems, see [Chapter 22 (Source Code Annotation and ATP Integration)](./22.source_code_annotation.md).

## 18.2 Template Declarations

### 18.2.1 Basic Form: `definition` Block with `let` Parameters

A template is a `definition` block whose leading `let` declarations establish the template parameters. All items in the block share their scope.

**Syntax (EBNF):**

```ebnf
template_def    ::= "definition"
                    { let_param }
                    { template_item }
                    "end" ";" ;

let_param       ::= "let" identifier_list "be" type_expression
                    [ "such" "that" formula ( "by" label_ref_list | proof ) ]
                    ";" ;

template_item   ::= attr_def
                  | mode_def
                  | struct_def
                  | pred_def
                  | func_def
                  | theorem_item
                  | registration_item ;
```

**Example — Parameterized magma structure:**

```mizar
definition
  let T be type;
  struct MagmaStr[T] where
    field carrier -> T;
    field binop -> BinOp of T;
  end;
end;
```

The parameterized name `MagmaStr[T]` uses `[...]` brackets to indicate that the name carries a type argument. This bracket notation is **reserved** for parameterized template names and scheme application; it must not be used for other purposes (such as list indexing). The `[...]` argument list may be omitted when the parameters are uniquely inferrable from context (see §18.2.7).

Automatic synonyms are generated where applicable:

- `Module over R` becomes a synonym for `Module[R]` (single struct parameter)
- `Subset of X` becomes a synonym for `Subset[X]` (single mode parameter)

### 18.2.2 Type Parameters

`let X be <type>;` declares a **type parameter** ranging over types (modes or structures).

| Declaration | Meaning |
|---|---|
| `let T be type;` | T is any type |
| `let G be Group;` | G must satisfy the `Group` mode |
| `let R be commutative Ring;` | R must be a commutative Ring |

```mizar
:: Single type parameter: polynomial ring over a commutative ring
definition
  let R be commutative Ring;
  struct PolyRing[R] where
    field carrier -> non empty set;   :: the underlying set of polynomials
    field add -> BinOp of carrier;
    field mul -> BinOp of carrier;
    field coeff -> Function of [:carrier, Nat:], R.carrier;
  end;
  :: Automatic synonym: PolyRing over R
end;

:: Two type parameters: generic homomorphism mode
definition
  let G, H be Group;
  mode Homomorphism[G,H] is
    operation_preserving Function of G.carrier, H.carrier;
  :: Automatic synonym: Homomorphism of G, H
end;
```

**Bounded type parameters**

`let T be type extends M` declares a **bounded type variable**: T ranges over any structure that inherits from M (via the `inherit ... from ...` chain), rather than a specific instance of M. When M is a mode, T is additionally required to satisfy all attributes that define M — for example, `let T be type extends commutative Magma` requires not only that T's struct inherits from Magma, but also that T satisfies the `commutative` attribute.

| Declaration | Meaning |
|---|---|
| `let T be type extends Group;` | T is any structure inheriting from Group |
| `let T be type extends commutative Magma;` | T bounded by commutative Magma |
| `let K be Field; let T be type extends VectorSpace[K];` | T bounded by parameterized mode |

Attribute forms are sugar for `such that` constraints: `let T be type extends commutative Magma` is equivalent to `let T be type extends Magma such that T is commutative`.

**Instantiation with `as` coercion**

When a **mode name** is provided directly, it must unambiguously identify the bound mode — write the intended mode name:

```mizar
PermProduct[AddMagma]            :: OK — mode name directly specifies additive Magma
PermProduct[commutative MulMagma]  :: OK — multiplicative Magma
PermProduct[Ring]                :: error — Ring reaches Magma via both AddGroup and MulMonoid
```

When a **variable** is provided whose structure has multiple inheritance paths to the bound mode, the caller must select the intended path with `V as M`:

```mizar
let R be commutative Ring;
PermProduct[R as AddMagma]   :: select R's additive Magma structure
PermProduct[R as MulMagma]   :: select R's multiplicative Magma structure
```

`V as M` is well-formed iff V's mode inherits from M via some `inherit ... from M` chain.

**Notation remapping**

Inside a template with `let T be type extends M`, M's notation applies. When instantiated via `as`, notation is remapped through the field mapping declared in the `inherit` chain:

```mizar
inherit AddGroup from Group where
  field add  from mul;
  field zero from unit;
  field neg  from inv;
end;
```

A theorem using `*` (Group's `mul` notation) applied as `[R as AddGroup]` for a specific ring variable R displays as `+` (AddGroup's `add` notation) at the use site.

**Example — permutation invariance of finite products:**

```mizar
definition
  let T be type extends commutative Magma;
  theorem PermProduct[T]:
    for s being FinSequence of T,
        p being Permutation of dom s
    holds Product[T](s) = Product[T](s * p)
  proof
    :: by induction on len s, using commutativity of T
    ...
  end;
end;

:: Application: sum permutation in any Ring (AddMagma directly — unambiguous)
PermProduct[AddMagma]

:: Application: product permutation in any commutative Ring
PermProduct[commutative MulMagma]

:: Application to a specific ring variable (as-coercion on the variable)
let R be commutative Ring;
PermProduct[R as AddMagma]   :: use R's additive structure
```

### 18.2.3 Predicate Parameters

`let P be pred(T₁, ..., Tₙ);` declares a **predicate parameter** ranging over n-ary predicates. Predicate parameters are valid only within `theorem` items in the same `definition` block.

Predicate parameters follow the `defpred` convention: applied as `P(t₁, ..., tₙ)` inside theorem bodies.

```mizar
definition
  let P be pred(Nat);
  theorem NatInduction[P]:
    P(0) & (for n being Nat st P(n) holds P(n+1))
    implies for n being Nat holds P(n)
  proof
    assume Base: P(0);
    assume Step: for n being Nat st P(n) holds P(n+1);
    let n be Nat;
    thus P(n) by Base, Step, mml.number.natural.Nat_induction;
  end;
end;
```

Schema theorems are ordinary implication theorems. Callers prove the antecedent conditions as named labels and cite them via `by NatInduction[P], Label1, Label2`.

### 18.2.4 Functor Parameters

`let F be func(T₁, ..., Tₙ) -> S;` declares a **functor parameter** ranging over functions of the given signature. Functor parameters are valid only within `theorem` items in the same `definition` block.

Functor parameters follow the `deffunc` convention: applied as `F(t)` inside theorem bodies.

```mizar
definition
  let A be type;
  let F be func(A) -> A;
  let P be pred(A);
  theorem ClosedUnderCompose[F,P]:
    (for x being A st P(x) holds P(F(x)))
    implies for x being A st P(x) holds P(F(F(x)))
  proof
    assume Step: for x being A st P(x) holds P(F(x));
    let x be A;
    assume Px: P(x);
    thus P(F(F(x))) by Px, Step;
  end;
end;
```

**Functor parameters vs first-order functions**

A functor parameter `let F be func(T) -> S;` is a **schema-level function symbol** — it is not a set in the domain of discourse, and does not support `is`-predicate reasoning. This distinguishes it from first-order functions defined via modes (e.g., `let phi be Homomorphism[G,H];`), which are set elements.

| Kind | Declaration | FOL status | `is` predicates |
|---|---|---|---|
| First-order function (mode-based) | `let phi be Homomorphism[G,H];` | Set element | Supported |
| Schema functor parameter | `let F be func(T) -> S;` | Meta-level symbol | Not supported |

When algebraic properties are needed on a function parameter, use a mode-based type declaration (`let phi be Homomorphism[G,H];`). Reserve `func(...)` parameters for schemas where no algebraic constraint is required.

### 18.2.5 Constrained Parameters

When a type parameter must satisfy an additional condition beyond its mode, state it with `such that`. The caller must supply evidence that the constraint holds for the specific instance used.

```mizar
definition
  let F be Field;
  let V be VectorSpace[F] such that V is finite_dimensional;
  :: Constraint: verified at each use site
  mode Basis[V] is finite linearly_independent Subset of V.carrier;
end;
```

At a use site, the caller must discharge the constraint as a named label:

```mizar
let F be REAL_FIELD;
let W be finite_dimensional VectorSpace[REAL_FIELD];
let phi be Homomorphism[W, W];
set V = image(phi);
reconsider V as VectorSpace[REAL_FIELD] by mml.algebra.linear.hom_image_vspace;
FinDim: V is finite_dimensional by mml.algebra.linear.image_fin_dim;
:: now Basis[V] is usable — constraint FinDim is discharged
let B be Basis[V] by FinDim;
```

**When `such that` is needed**: Use it when a **definition item** (mode, attr, functor, structure) in the block requires the constraint for its well-formedness. It is **not** appropriate when only a theorem uses the condition — in that case, write an ordinary implication hypothesis instead:

```mizar
:: Preferred: express finite-dimensionality as a theorem hypothesis
theorem BasisCard:
  for F being Field, V being VectorSpace[F] st V is finite_dimensional
  holds card (Basis of V) = dim V;
:: No template needed — no definition item depends on the constraint
```


### 18.2.6 Scoping Rules

- `let` declarations at the beginning of a `definition` block scope over **all items** in the block. This follows the classical Mizar scoping rule for `let IT be ...;` in definition blocks.
- Predicate parameters (`let P be pred(T);`) and functor parameters (`let F be func(T) -> S;`) scope only to individual `theorem` items within the block; they may not be used in `attr`, `mode`, `struct`, `func`, or `pred` items.
- Parameterized names `N[params]` are visible within the enclosing **module** after the closing `end;`. They are not accessible from other modules without an explicit `import`.
- A `definition` block with parameterized definitions may not be nested inside another `definition` block or inside a `proof` block.
- Circular dependencies between parameterized definitions are forbidden.

### 18.2.7 Template Parameter Inference

When a template is applied, the `[...]` argument list may be **omitted** if all template parameters can be **uniquely inferred** from the types of the other arguments. This rule applies to `func` and `pred` templates.

```mizar
:: Explicit form — always valid
Product[Ring](s)
a +[Ring] b
s are_congruent[H]

:: Implicit form — valid when type is uniquely determined
Product(s)           :: T inferred from the element type of s
a + b                :: resolved from the types of a and b
s are_congruent(H)   :: H's type determines the pred
```

**Inference rule**: If the `[T]` arguments are omitted, the elaborator determines the parameters from the types of the remaining arguments. If the parameters can be uniquely determined, implicit instantiation succeeds. If two or more instantiations match, an ambiguity error is reported and explicit `[T]` is required.

```mizar
:: Ambiguous — error, explicit required
PermProduct[Ring]       :: Ring reaches Magma via AddGroup and via MulMonoid

:: Unambiguous — AddMagma directly names the desired mode
PermProduct[AddMagma]   :: OK

:: When a specific ring variable is in scope, as-coercion on the variable
let R be commutative Ring;
PermProduct[R as AddMagma]   :: use R's additive structure specifically
```

**`as` coercion is always explicit**: The `T as M` coercion (§18.2.2) is never inferred — it must be written explicitly when applied to a specific variable (e.g., `R as AddMagma`) to select one of its multiple inheritance paths. When the mode name itself is unambiguous (e.g., `AddMagma`), no `as` coercion is needed.

The inference mechanism extends the overloading resolution already used for classical Mizar operator definitions uniformly to the full template system.

## 18.3 Template for Attributes (attr)

Attribute templates define families of attributes indexed by type parameters. The template parameter must be **independent** of the subject's type — it carries information not already determined by the subject itself.

**Relationship to Chapter 6 attribute syntax**

The hyphenated attribute form from Chapter 6 is syntactic sugar for the single-parameter template form, following the rule `G-attrname` ↔ `attrname[G]`:

| Sugar form (Chapter 6) | Template form |
|---|---|
| `n is G-valid_exponent` | `n is valid_exponent[G]` |
| `S is M-dense` | `S is dense[M]` |

Both forms are accepted. The `[G]` form is required for multi-parameter attributes.

**Example 1 — Exponent of a group:**

```mizar
definition
  let G be Group;
  let n be positive Nat;
  attr ExponentDef: n is valid_exponent[G] means
    for g being Element of G holds g^n = G.unit;
  :: Automatic synonym: n is G-valid_exponent
end;
```

`g^n = G.unit` means every element of G raised to the n-th power equals the identity.

**Usage:**

```mizar
let G be finite Group;
assume 12 is valid_exponent[G];
:: equivalently: assume 12 is G-valid_exponent
```

**Example 2 — Dense subset in a metric space:**

```mizar
definition
  let M be MetricSpace;
  let S be Subset of M.carrier;
  attr DenseDef: S is dense[M] means
    for x being Element of M, eps being positive Real holds
      ex s being Element of S st dist_M(x, s) < eps;
  :: Automatic synonym: S is M-dense
end;
```

**Usage:**

```mizar
let M be MetricSpace;
let S be Subset of M.carrier;
assume S is dense[M];
:: equivalently: assume S is M-dense
```

## 18.4 Template for Modes (mode)

Mode templates define families of types indexed by type parameters, generalizing the `of`/`over` parameterization.

**Example 1 — Generic subobject mode:**

```mizar
definition
  let A be type;
  mode Subobject[A] is A-subset_like set;
  :: Automatic synonym: Subobject of A
end;

:: Usage
let S be Subobject[G];
:: equivalently: let S be Subobject of G;
```

**Example 2 — Generic homomorphism mode (two parameters):**

```mizar
definition
  let G,H be Group;
  mode Homomorphism[G,H] is
    operation_preserving Function of G.carrier, H.carrier;
  :: Automatic synonym: Homomorphism of G, H
end;

:: Usage
let phi be Homomorphism[G, H];
```

**Example 3 — Constrained mode:**

```mizar
definition
  let F be Field;
  let V be VectorSpace[F] such that V is finite_dimensional;
  mode FinBasis[V] is finite linearly_independent Subset of V.carrier;
  :: FinBasis[V] only makes sense for finite-dimensional V
end;
```

At a use site, the caller must discharge the constraint:

```mizar
let F be REAL_FIELD;
let V be VectorSpace[REAL_FIELD];
FinDim: V is finite_dimensional proof ... end;
let B be FinBasis[V] by FinDim;  :: valid — FinDim discharges the constraint
```

## 18.5 Template for Structures (struct)

Structure templates define parameterized composite types.

**Example 1 — Generic vector space over a field:**

```mizar
definition
  let K be Field;
  struct VectorSpace[K] where
    field carrier -> non empty set;
    field add -> BinOp of carrier;
    field smul -> Function of [:K.carrier, carrier:], carrier;
  end;
  :: Automatic synonym: VectorSpace over K
end;

:: Usage: instantiated for real numbers
let V be VectorSpace[REAL_FIELD];
:: equivalently: let V be VectorSpace over REAL_FIELD;
```

**Example 2 — Generic ordered structure with inheritance:**

```mizar
definition
  let T be set;
  struct OrdStr[T] where
    field carrier -> T;
    field order -> Relation of T;
  end;
  :: Automatic synonym: OrdStr over T

  inherit OrdStr[T] from 1-sorted where
    field carrier from carrier;
  end;
end;
```

**Example 3 — Structure with multiple type parameters:**

```mizar
definition
  let F be Field;
  let V,W be VectorSpace[F];
  struct BilinearMap[F,V,W] where
    field map -> Function of [:V.carrier, W.carrier:], F.carrier;
    property kernel -> Subobject[V];
  end;
  :: Automatic synonym: BilinearMap over F, V, W
end;

:: Usage
let B be BilinearMap[REAL_FIELD, V, W];
```

## 18.6 Template for Predicates (pred)

Predicate templates define families of predicates indexed by type parameters.

**Example 1 — Generic congruence predicate:**

```mizar
definition
  let G be Group;
  let H be Subgroup of G;
  let a, b be Element of G;
  pred GroupCongruencePred: a, b are_congruent[H] means
    a * b" in H;
end;

:: Usage
let G be Group;
let H be Subgroup of G;
let a, b be Element of G;
assume a, b are_congruent[H];
```

**Example 2 — Congruence modulo an ideal:**

```mizar
definition
  let R be commutative Ring;
  let I be Ideal of R;
  let a, b be Element of R;
  pred RingCongruencePred: a, b are_congruent_mod[I] means
    a - b in I.carrier;
  reflexivity
  proof
    let a be Element of R;
    thus a - a in I.carrier by mml.algebra.ring.Ring_sub_self_zero, I.zero_in_ideal;
  end;
  symmetry
  proof
    let a, b be Element of R;
    assume a - b in I.carrier;
    thus b - a in I.carrier by mml.algebra.ring.Ring_neg_closed, I.ideal_neg_closed;
  end;
  transitivity
  proof
    let a, b, c be Element of R;
    assume a - b in I.carrier;
    assume b - c in I.carrier;
    thus a - c in I.carrier by mml.algebra.ring.Ring_add_closed, I.ideal_add_closed;
  end;
end;
```

## 18.7 Template for Functors (func)

Functor templates define families of operations parameterized by type parameters.

**Example 1 — Generic identity function:**

```mizar
definition
  let T be type;
  func IdDef: id[T] -> Function of T, T equals
    { [x, x] where x is Element of T : x in T };
end;

:: Usage
let f = id[NAT];
```

**Example 2 — Generic neutral element:**

```mizar
definition
  let G be Group;
  func UnitDef: unit[G] -> Element of G means
    for a being Element of G holds it * a = a & a * it = a;
  existence
  proof
    consider e being Element of G such that
      A1: for a being Element of G holds e * a = a & a * e = a
      by mml.algebra.group.Group_identity_exists;
    take e;
    thus thesis by A1;
  end;
  uniqueness
  proof
    let e1, e2 be Element of G;
    assume A1: for a being Element of G holds e1 * a = a & a * e1 = a;
    assume A2: for a being Element of G holds e2 * a = a & a * e2 = a;
    thus e1 = e1 * e2 by A2
           .= e2 by A1;
  end;
end;
```

## 18.8 Parameterized Theorems and Schemes

Theorems whose parameters are **sets** (including sets constrained by a mode, such as `Group` or `Ring`) can be written as ordinary universally quantified theorems — `for G being Group` is first-order since it is equivalent to `for G being set st is_Group(G)`:

```mizar
theorem CommMul:
  for G being commutative Group, a, b being Element of G holds a * b = b * a;
```

Templates are required when a theorem is parameterized by:
- a **type** (`let T be type`) — `for T being type` is **not permitted** in ordinary theorem statements;
- a **predicate** (`let P be pred(T)`) — scheme-style parameterization;
- a **functor** (`let F be func(T) -> S`) — parameterization by a function symbol.

### 18.8.1 Type-Parameterized Theorems

A theorem co-defined with parameterized definitions in a `definition` block can share the block's `let T be type` parameter. This is also the required form for theorems that range over an abstract type.

```mizar
definition
  let T be type;
  theorem FinSeqConcat[T]:
    for s, t being FinSequence[T]
    holds len (s ^ t) = len s + len t
  proof
    ...
  end;
end;
```

### 18.8.2 Predicate-Parameterized Theorems (Schemes)

A theorem with `let P be pred(T);` in the enclosing `definition` block is a **predicate schema** — it generates a family of first-order theorems, one per predicate substituted for `P`. Antecedent conditions are introduced with `assume Label: ...` inside the proof.

```mizar
definition
  let P be pred(Nat);
  theorem NatInduction[P]:
    P(0) & (for n being Nat st P(n) holds P(n+1))
    implies for n being Nat holds P(n)
  proof
    assume Base: P(0);
    assume Step: for n being Nat st P(n) holds P(n+1);
    let n be Nat;
    thus P(n) by Base, Step, mml.number.natural.Nat_induction;
  end;
end;
```

### 18.8.3 Functor-Parameterized Theorems

A theorem with `let F be func(T) -> S;` is parameterized by a **schema-level function symbol** treated as a black box. When algebraic properties are needed on the function, use a mode-based parameter instead (see §18.2.4).

The following theorem states: if `F` is monotone with respect to `P`, and `P(x, F(x))`, then `P(F(x), F(F(x)))`. This is a basic step in the theory of ascending chains and fixed points, and holds for any `F` and any order `P`.

```mizar
definition
  let A be type;
  let P be pred(A, A);
  let F be func(A) -> A;
  theorem MonotoneStep[P,F]:
    (for x, y being A st P(x, y) holds P(F(x), F(y)))
    implies for x being A st P(x, F(x)) holds P(F(x), F(F(x)))
  proof
    assume Mono: for x, y being A st P(x, y) holds P(F(x), F(y));
    let x be A;
    assume P(x, F(x));
    thus P(F(x), F(F(x))) by Mono;
  end;
end;
```

## 18.9 Scheme Application in Proofs

A predicate or functor schema is applied in a proof by:

1. Defining the predicate argument with `defpred` (or the functor argument with `deffunc`).
2. Proving the antecedent conditions of the implication as named labels in the current proof context.
3. Referencing the schema theorem with `by SchemaName[P], Label1, Label2, ...`.

**Syntax (EBNF):**

```ebnf
scheme_app  ::= "by" theorem_label "[" scheme_arg { "," scheme_arg } "]"
                { "," label_reference } ";" ;

scheme_arg  ::= defpred_identifier | deffunc_identifier ;
```

> [!IMPORTANT]
> Inline λ-style predicate arguments (e.g., `by NatInduction[pred n means n * 2 is even]`) are **not** supported. Always introduce a `defpred` (or `deffunc`) before applying a schema. This promotes readability and allows the verifier to resolve the predicate definition unambiguously.

**Example — Applying natural number induction:**

```mizar
theorem NatMulEven:
  for n being Nat holds n * 2 is even
proof
  defpred P(n) means n * 2 is even;
  Base: P(0)
  proof
    thus 0 * 2 is even by mml.number.nat.zero_even;
  end;
  Step: for n being Nat st P(n) holds P(n+1)
  proof
    let n be Nat;
    assume A1: n * 2 is even;
    n * 2 + 2 = (n + 1) * 2 by mml.number.nat.distrib;
    thus (n + 1) * 2 is even by A1, mml.number.nat.even_plus_even;
  end;
  thus for n being Nat holds P(n) by NatInduction[P], Base, Step;
end;
```

## 18.10 Logic Encoding Details

Templates are meta-level constructs. Each **use site** (where a parameterized name is used or a schema is applied) generates a collection of first-order axioms and proof obligations.

### 18.10.1 Templates as Axiom Schemas

A `definition` block with `let` parameters is treated as an **axiom schema** — a recipe for generating FOL axioms when a parameterized name is used or a schema theorem is referenced.

| Template Level | FOL Status |
|---|---|
| `definition` block with `let` parameters | No FOL axioms generated (schema only) |
| Use of parameterized name (e.g., `let M be Module[R];`) | Generates axioms for that instantiation of `R` |
| Schema application (`by NatInduction[P], Base, Step`) | Generates the specific first-order instance |

**Verification guarantee**: The verifier checks each template body **once symbolically** at declaration time. All proof obligations in the body are discharged relative to the parameter constraints. This ensures that every use generates valid axioms without re-proving.

### 18.10.2 Type Parameter Encoding

A type parameter `let T be type;` is encoded as an **uninterpreted predicate symbol** `is_T(x)` in the schema.

| Source | FOL Schema |
|---|---|
| `let T be type;` | Schema variable `is_T/1` (uninterpreted) |
| `let G be Group;` | Schema variable `is_G/1` with guard `∀x. is_G(x) → is_Group(x)` |
| `let T be type extends M;` | Schema variable `is_T/1` with guard `∀x. is_T(x) → is_M(x)` |
| `let x be T;` | Guard `is_T(x)` added as antecedent |
| `v as N` at use site (v a variable, N ancestor of v's bound) | No new axiom; `is_T(v) → is_M(v) → is_N(v)` by inheritance chain; selects N-structure of v |

**Example:**

FOL schema for `Homomorphism[G, H]` with `G, H` being `Group`:

```
∀G, H. (∀x. is_G(x) → is_Group(x)) ∧ (∀x. is_H(x) → is_Group(x))
  → ∀f. (is_Hom(f, G, H) ↔ is_operation_preserving(f, G, H))
```

For usage `G := INT_GROUP`, `H := REAL_GROUP`:

```
∀f. (is_Hom(f, INT_GROUP, REAL_GROUP)
       ↔ is_operation_preserving(f, INT_GROUP, REAL_GROUP))
```

### 18.10.3 Predicate Parameter Encoding

A predicate parameter `let P be pred(T);` is encoded as an **uninterpreted predicate symbol** `P/1`. The schema represents a **family of first-order formulas** — one for each predicate substituted for `P`. Since FOL does not support second-order quantification, each instantiation generates a distinct first-order axiom.

| Source | FOL Schema |
|---|---|
| `let P be pred(T);` | Schema variable `P/1` (uninterpreted) |
| `P(x)` in formula | `P(x)` in generated formula |
| `for x st P(x) holds Q` | `∀x. (P(x) → Q)` in generated formula |

**Example — NatInduction schema:**

FOL schema (one instance for each predicate `P` substituted):

```
P(0) ∧ (∀n. is_Nat(n) → P(n) → P(succ(n)))
  → ∀n. is_Nat(n) → P(n)
```

This is precisely the **natural number induction axiom schema** in FOL. The trusted kernel verifies each specific instance (e.g., `P(n) := is_even(n * 2)`) without requiring higher-order reasoning.

### 18.10.4 Functor Parameter Encoding

A functor parameter `let F be func(T) -> S;` is encoded as an **uninterpreted function symbol** `F/1` with a type guard axiom.

| Source | FOL Schema |
|---|---|
| `let F be func(T) -> S;` | Uninterpreted function `F/1` with guard `∀x. is_T(x) → is_S(F(x))` |
| `F(x)` in term | `F(x)` |

**Example — `ClosedUnderCompose` with func parameter:**

FOL schema for `ClosedUnderCompose[F,P]` (§18.2.4), where `F` is a schema functor and `P` is a schema predicate:

```
(∀x. is_A(x) ∧ P(x) → is_A(F(x)))          (* F: A → A guard *)
∧ (∀x. is_A(x) ∧ P(x) → P(F(x)))            (* antecedent: F preserves P *)
  → ∀x. is_A(x) ∧ P(x) → P(F(F(x)))         (* conclusion *)
```

Each concrete instantiation (e.g., `F := succ`, `P(n) := n is even`) produces a distinct first-order axiom.

### 18.10.5 Constraint Encoding

Parameter constraints (`such that` clause) are encoded as **conditional guards** — proof obligations deferred to use sites.

| Source | FOL Effect |
|---|---|
| `let V be T such that φ` | Constraint `φ` must be proved at each use site; acts as antecedent in all generated axioms |
| Hypothesis in `implies` clause | Antecedent in the generated FOL implication |

**Example — `such that` for a definition item:**

```mizar
definition
  let F be Field;
  let V be VectorSpace[F] such that V is finite_dimensional;
  mode FinBasis[V] is finite linearly_independent Subset of V.carrier;
end;
```

Generated axiom at use site `F := F₀`, `V := V₀` (after caller proves `V₀ is finite_dimensional`):

```
is_Field(F₀) ∧ is_VectorSpace(V₀, F₀) ∧ is_finite_dimensional(V₀)
  → ∀x. (is_FinBasis(x, V₀) ↔ is_finite(x) ∧ is_linearly_independent(x, V₀))
```

**Example — `implies` for a theorem hypothesis:**

```mizar
theorem BasisCard:
  for F being Field, V being VectorSpace[F] st V is finite_dimensional
  holds card (Basis of V) = dim V;
```

FOL encoding (no template needed — ordinary universal quantification):

```
∀F, V. (is_Field(F) ∧ is_VectorSpace(V, F) ∧ is_finite_dimensional(V)
         → card(Basis(V)) = dim(V))
```

## 18.11 Complete Syntax (EBNF)

```ebnf
(* === Template Declaration (definition block with let parameters) === *)
template_def    ::= "definition"
                    { let_param }
                    { template_item }
                    "end" ";" ;

let_param       ::= "let" identifier_list "be" let_type
                    [ "such" "that" formula
                      ( "by" label_ref_list | proof ) ]
                    ";" ;

(* === Type in let_param: plain type expression, or abstract type with optional bound === *)
let_type        ::= "type" [ "extends" bound_type ]
                  | type_expression ;

bound_type      ::= [ attribute_chain ] radix_type ;

template_item   ::= attr_def
                  | mode_def
                  | struct_def
                  | pred_def
                  | func_def
                  | theorem_item
                  | registration_item ;

(* === Predicate Parameters (in let_param; valid in theorem items only) === *)
pred_param      ::= identifier "be" "pred" "(" type_list ")" ;

type_list       ::= type_expression { "," type_expression } ;

(* === Functor Parameters (in let_param; valid in theorem items only) === *)
func_param      ::= identifier "be" "func" "(" type_list ")" "->" type_expression ;

(* === Parameterized Names (reserved bracket notation) === *)
(* The "[" type_arg_list "]" part may be omitted when parameters are uniquely
   inferrable from the argument types; see §18.2.7. *)
param_name      ::= identifier [ "[" type_arg_list "]" ] ;
type_arg_list   ::= type_arg { "," type_arg } ;
type_arg        ::= type_expression | coercion_arg ;
coercion_arg    ::= identifier "as" radix_type ;
    (* identifier must be a variable (not a mode name); well-formed iff
       the variable's bound mode inherits from radix_type via inherit chain *)

(* === Theorem Items with Predicate/Functor Parameters === *)
(* Schema theorems are ordinary implication theorems:
   hypothesis conditions appear as antecedents of the formula itself. *)
theorem_item    ::= ( "theorem" | "conjecture" ) label_identifier ":"
                    formula
                    [ justification ] ";" ;

(* === Scheme Application in Proofs === *)
(* scheme_name is the label of a schema theorem (e.g. NatInduction).
   Arguments are passed via template_args (defined in Chapter 16 §16.5):
     - For predicate/functor parameters: defpred_identifier | deffunc_identifier
     - For type parameters: type_expression | coercion_arg
   Additional proof-local labels follow after the template_args. *)
scheme_app      ::= "by" scheme_name template_args
                    { "," label_reference } ";" ;
    (* template_args  ::=  "[" template_arg { "," template_arg } "]"
       template_arg   ::=  type_expression | coercion_arg
                         | defpred_identifier | deffunc_identifier
       — see Chapter 16 §16.5 for the full definition *)

scheme_name     ::= label_identifier ;

(* === Predicate/Functor Application (following defpred/deffunc convention) === *)
pred_app        ::= identifier "(" term_list ")" ;
func_app        ::= identifier "(" term_list ")" ;
```
